# Big-O Notation

Big-O notation describes how an algorithm's runtime or space usage grows as the input size increases. It gives us a way to compare algorithms independent of hardware or implementation details.

## Why it matters

When choosing between algorithms, we need a way to predict performance at scale. An algorithm that works fine on 100 items might be unusable on 1,000,000 items. Big-O tells us **the growth rate** — not the exact time.

## Common complexities

| Notation | Name | Example |
|----------|------|---------|
| $O(1)$ | Constant | Array access by index |
| $O(\log n)$ | Logarithmic | Binary search |
| $O(n)$ | Linear | Single loop through array |
| $O(n \log n)$ | Linearithmic | Merge sort, quick sort (average) |
| $O(n^2)$ | Quadratic | Nested loops (bubble sort) |
| $O(2^n)$ | Exponential | Recursive fibonacci (naive) |
| $O(n!)$ | Factorial | Generating all permutations |

## Rules for calculating Big-O

### 1. Drop constants

$O(2n) = O(n)$, $O(100) = O(1)$

We care about growth rate, not exact operations. Whether your loop does 1 or 5 operations per element, it's still $O(n)$.

### 2. Drop lower-order terms

$O(n^2 + n) = O(n^2)$

As $n$ grows large, the dominant term determines the growth rate.

### 3. Different inputs → different variables

If a function takes two arrays of different sizes, use different variables:

```python
def print_pairs(a, b):
    for x in a:       # O(a)
        for y in b:   # O(b)
            print(x, y)
# Total: O(a × b), NOT O(n²)
```

        else:

## Space complexity

Big-O also applies to **memory usage**. Common patterns:

| Pattern | Space | Example |
|---------|-------|---------|
| Fixed variables | $O(1)$ | Swapping two numbers |
| Copy of input | $O(n)$ | Creating a sorted copy |
| 2D matrix | $O(n^2)$ | Adjacency matrix |
| Recursion stack | $O(n)$ or $O(\log n)$ | Depends on recursion depth |

    original = [3, 1, 4, 1, 5, 9, 2, 6]

    print(f"Unique elements: {seen}")

## Best, Average, and Worst Case

An algorithm can have different complexities depending on the input:

- **Best case**: The most favorable input (e.g., already sorted for insertion sort → $O(n)$)
- **Average case**: Expected performance for random input
- **Worst case**: The least favorable input (e.g., reverse sorted for insertion sort → $O(n^2)$)

Big-O typically refers to the **worst case** unless stated otherwise.

## Amortized analysis

Some operations are usually fast but occasionally slow. **Amortized analysis** averages the cost over a sequence of operations.

**Example: Dynamic array (Python list)**
- Most `append()` calls are $O(1)$
- Occasionally, the array must resize (copy all elements) — $O(n)$
- Amortized cost: still $O(1)$ per append, because resizing happens rarely enough

---

## Key takeaways

1. Big-O describes **growth rate**, not exact time
2. Focus on the **dominant term** — drop constants and lower-order terms
3. Different inputs need **different variables**
4. Consider both **time and space** complexity
5. Worst case is the default unless stated otherwise
